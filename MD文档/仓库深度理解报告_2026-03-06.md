# 仓库深度理解报告（代码结构 / 训练链路 / 风险点）

> 目的：仅用于沉淀“对现有工程的深入理解”，不引入功能变更。

## 1. 项目目标与当前阶段定位

该仓库核心任务是**视网膜血管分割**，以 U-Net（`segmentation_models_pytorch.Unet + resnet34 encoder`）为主干，并在基线上引入持续同调拓扑约束以改善连通性问题。工程中并行保留两条训练线：

- 基线线：`train_baseline.py`（Dice Loss only）
- 拓扑线：`train_with_topology.py`（Dice + λ·Topo）

并通过 `config.yaml` 统一控制数据模式（纯 DRIVE / Kaggle 联合）、训练超参、拓扑损失参数和 λ 调度策略。

---

## 2. 代码主干与职责边界

### 2.1 数据层

- `data_drive.py`
  - 面向原始 DRIVE 目录结构，固定 16+4 划分（训练 21-36，验证 37-40），测试集 01-20。
  - 训练样本返回 `(image, vessel_mask, roi_mask)`，测试无血管标签时以全零占位。
  - 关键防线：对训练标签均值做异常告警，避免误把 ROI 当血管标签。

- `data_combined.py`
  - 当前训练入口统一经由该文件的 `get_combined_loaders()`。
  - `use_kaggle_combined=false`：透传到 DRIVE loader；`true`：走 Kaggle 联合集逻辑。
  - Kaggle 路径优先本地 `data/combined`，否则 `kagglehub.dataset_download()` 自动拉取。
  - `KaggleCombinedDataset` 提供：鲁棒 mask 匹配、等比缩放+padding、RGB 归一化、可选 FOV 椭圆 ROI 推断、同步增强。

### 2.2 模型层

- `model_unet.py`
  - 统一通过 `get_unet_model()` 创建 U-Net，强制 `encoder_weights=None` 防网络下载。
  - 若 `pretrained=true`，通过 `load_local_resnet34_weights()` 手动注入本地权重并打印匹配统计。
  - 提供 `load_model()` 按 checkpoint 恢复模型（推理评估使用）。

### 2.3 损失层

- `topology_loss.py`
  - `TopologicalRegularizer` 使用 `cripser.compute_ph_torch` 做可微 0 维持续同调。
  - 先将预测概率映射为子水平集 filtration（`1-prob`），提取 finite lifetimes，取 top-k（`target_beta0`），再做 `mse` 或 `hinge`。
  - 返回损失时乘 `loss_scale`，训练端再乘 λ。

### 2.4 指标层

- `utils_metrics.py`
  - 基础指标：Dice / IoU / Precision / Recall（可 ROI 约束）。
  - 拓扑指标：骨架碎片数 `CL-Break` + 过滤后连通分量差 `Δβ₀`。
  - `compute_topology_metrics()` 失败时返回 -1，避免流程中断。

### 2.5 训练与评估层

- `train_baseline.py`
  - 使用 `get_combined_loaders()` 读取数据，`DiceLoss(from_logits=True)` 训练。
  - 验证阶段逐样本统计基础+拓扑指标（拓扑仅日志监控）。
  - 支持早停开关，但当前配置默认关闭。

- `train_with_topology.py`
  - 同样走 `get_combined_loaders()`。
  - 使用 `LambdaScheduler`（策略 0 / 015 / 3175）调度拓扑权重。
  - 训练日志额外记录 `dice_loss/topo_raw/topo_scaled/ratio/ROI统计`，更便于排查拓扑项是否真实生效。

- `evaluate.py`
  - 目前评估路径绑定 `data_drive.get_drive_loaders()`，与训练阶段“统一走 combined loader”存在接口分叉。
  - 可保存 `npy` 预测并输出可视化图。

## 3. 关键数据流（从输入到指标）

1. DataLoader 输出 `image[3,H,W], vessel[1,H,W], roi[1,H,W]`。  
2. 模型输出 logits `[B,1,H,W]`。  
3. 训练损失：
   - Baseline：`L = Dice(logits, vessel)`
   - Topology：`L = Dice + λ * Topo(sigmoid(logits), roi)`
4. 验证与评估：概率图阈值化后在 ROI 内计算基础指标；骨架化后得 `CL-Break`，连通域过滤后得 `Δβ₀`。

---

## 4. 当前配置语义（config.yaml）

- 数据模式默认：`use_kaggle_combined: true`（即训练默认走联合集）。
- ROI 策略：`kaggle_roi.mode: fov`，并非全 1；对应 `threshold/padding_scale` 控制 FOV 椭圆拟合。
- 模型输入：工程实际统一 RGB 3 通道。
- 拓扑配置：`target_beta0=5, loss_scale=100.0, lambda_schedule.strategy="3175"`。
- 训练控制：`enable_early_stopping=false`，将跑满 `max_epochs`。

---

## 5. 设计亮点

- **双数据模式集中在一处切换**：训练入口不分叉，减少脚本层条件逻辑复杂度。
- **本地预训练权重注入机制完善**：可离线训练，且加载统计可审计。
- **拓扑日志细粒度**：`ratio` 与 `roi_all_ones` 等字段可快速定位“拓扑项虚设”问题。
- **Kaggle ROI 可配置**：支持 `ones/fov/tiny`，兼顾对照实验与鲁棒训练。

---

## 6. 需要重点关注的潜在一致性问题（理解层结论）

1. **训练与评估数据入口不一致**：训练走 `data_combined`，评估脚本仍直连 `data_drive`，在 Kaggle 模式下会造成流程割裂。  
2. **文档与代码可能存在时序偏差**：仓库内部分审计文档仍描述“联合集 ROI 为全1”，但当前配置已设为 `fov`。  
3. **拓扑损失参数名语义差异**：`max_death` 在当前实现中主要做参数保留与打印，实际 forward 未显式裁剪 death。  
4. **归一化与增强次序需要实验确认**：当前为 `ToTensor -> Normalize -> Augment(含亮度/对比度)` 或变体，存在与常见流程不同的实现细节。

---

## 7. 结论

该仓库已经形成“可训练 + 可评估 + 可审计”的完整实验骨架，且在拓扑方向具备工程化可追踪性（调度、loss拆解、ROI可视化调试）。当前最关键的理解结论是：

- **主训练链路是稳定的**；
- **最值得后续治理的是跨脚本一致性（尤其 evaluate 与 combined loader 对齐）**；
- **项目文档较多，需以代码与 config 为最终事实源。**
