## 1. 你的方法 vs Hu 2019 的根本差异（必须先搞清楚）

> [!IMPORTANT]
> 以下是理解"应该借什么"的前提。很多同学读了 Hu 2019 后直接照搬实现，结果和自己的主线冲突——你必须理解两者的根本区别。

| 维度 | 你的当前方法 (Fragment-Suppress) | Hu et al. 2019 (TopoLoss) |
|---|---|---|
| **PH 计算后端** | `cripser` (cubical ripser, Python/PyTorch 绑定) | 自研 C++ persistence 库 (pybind11) |
| **Filtration 对象** | 预测概率图 `1 - prob`（仅预测端） | 预测端 **与 GT 端 双向** 计算 PD |
| **拓扑目标来源** | **无 GT 拓扑参考**——用先验规则（"只保留最长条，惩罚其余"）| **有 GT 拓扑参考**——计算 GT 的 PD，做 Wasserstein 匹配 |
| **Loss 公式** | `fragments[1:].pow(2).mean()` | `Σ ‖(b,d)_pred - matched_(b,d)_gt‖²` + 未匹配点到对角线距离 |
| **梯度作用方式** | 作用于所有碎片的 birth/death pixels（通过 cripser 自动微分） | 仅作用于 **critical pixels**（birth pixel 和 death pixel） |
| **哲学差异** | **正则化范式**——用拓扑先验约束输出 | **监督范式**——用 GT 拓扑指导输出 |

[External] 以上 Hu 方法的描述基于 NeurIPS 论文原文和 GitHub 仓库。

**关键判断 [Inference]**：你和 Hu 走的是 PH-for-segmentation 的 **两个不同分支**：
- Hu 是 **"预测 PD vs GT PD 的监督匹配"**
- 你是 **"无需 GT PD 的先验正则化"**

这个区别决定了你 **不应该照搬 Hu 的 loss 公式**，但 **应该在论文中清晰定位自己与 Hu 的关系**。

## 2. 逐篇文献三层分析

### 2.1 Hu et al. 2019 — Topology-Preserving Deep Image Segmentation (NeurIPS)

#### 方法思想层 ✅ 借

| 判定 | 理由 |
|---|---|
| **应该借** | 这是 PH-for-segmentation 的开创工作。你的整个 "把持续同调引入损失函数" 的想法追溯到这里。论文必须引，且作为 **方法论上游** 定位 |

**具体借什么**：
- 引用它建立 "PH 可用于分割损失函数" 的合法性
- 引用它的 "topological loss 可微" 结论来支撑你用 cripser 做自动微分的可行性
- 借它的问题定义框架："像素级损失忽略全局拓扑结构"

**不应该借什么**：
- 不要声称你的方法 "借鉴了 Hu 的实现" 或 "改进了 Hu 的方法"——你们用的是不同的 PH 后端和不同范式
- 不要暗示 Hu 的方法有问题——你要做的是"在 Hu 的思想基础上，探索了另一种映射策略"

#### 损失构造层 ⚠️ 可对比，不可照搬

| 判定 | 理由 |
|---|---|
| **可作对比分析，但不适合直接替换你的 loss** | Hu 用的是 Pred-PD vs GT-PD 的 Wasserstein 距离，需要对 GT 也计算 PH。你的框架没有这条路径 |

**如果要落地的最小改动** [Inference]：
1. 对 GT mask 也跑 `cripser.compute_ph_torch(1 - gt, maxdim=0)`
2. 提取 GT 的 0-dim persistence pairs
3. 在 pred PD 和 GT PD 之间做匹配（需实现匈牙利匹配或贪心匹配）
4. 用匹配距离代替 `fragments.pow(2).mean()`

**风险评估**：
- 工程量 **中等偏高**（匹配算法 + GT PD 缓存 + 调试）
- 训练速度 **会变慢**（每个样本需要双倍 PH 计算）
- 效果 **不确定**（GT PD 对视网膜血管树可能噪声很大——GT 标注本身就有碎片）
- **在你当前阶段（论文定稿前），风险远大于收益**

#### 工程实现层 ❌ 不要借

| 判定 | 理由 |
|---|---|
| **不适合** | Hu 的实现依赖自研 C++ PH 库（94.7% C++ 代码），需要 pybind11 编译。你已经用 cripser，替换没有意义 |

### 2.2 Shit et al. 2021 — clDice (CVPR)

> 你没有提到这篇，但它是当前面临的 **Level B 风险（缺少横向对比）** 的最佳解药。

#### 方法思想层 ✅ 应该在相关工作中讨论

| 判定 | 理由 |
|---|---|
| **必须引** | clDice 是视网膜血管分割中 **最被广泛使用的拓扑感知损失**，评委大概率知道这篇。如果你不讨论它，会被问 "为什么不用 clDice？" |

**具体借什么**：
- 在相关工作中作为 **另一类拓扑感知方法**（基于形态学骨骼 vs 基于持续同调）
- 用它帮你定位自己的位置："clDice 从空间结构（中心线）入手，本文从代数拓扑（PH）入手，两者互补"

#### 损失构造层 ⚠️ 可作为评估指标补充，不适合替换主线 loss

| 判定 | 理由 |
|---|---|
| **作为评估指标有价值，但不作为训练 loss** | 你的论文核心贡献是 PH 路线的诊断和修正。如果换成 clDice 做 loss，就偏离了主线 |

**最小可行改动（作为评估指标）** [Inference]：
- `pip install clDice` 或从 GitHub 移植 soft-skeleton 计算
- 在评估阶段对 Baseline 和 Topo-FS 的预测结果计算 clDice 指标
- 如果 Topo-FS 的 clDice 高于 Baseline，那就是一个 **有力的补充证据**

**风险评估**：
- 工程量 **低**（仅加一个评估指标）
- 如果 clDice 在 Topo-FS 上反而更低，会成为负面证据
- **适合作为论文的补充分析，不适合作为主线改动**

#### 工程实现层 ⚠️ 只借评估部分

| 判定 | 理由 |
|---|---|
| **评估可借，训练不借** | 训练端移植需要 soft-skeleton 实现，和你的 PH 主线完全无关 |

## 4. What to Borrow Next, in One Week — 极简计划

### Day 1–2：论文相关工作章节起草

**做什么**：用 Hu 2019 建立你的论文核心引用框架。

写三段话：
1. **Hu 2019 做了什么**（2-3 句）：首次提出将 PH 直接应用于分割损失函数，通过比较 pred PD 和 GT PD 的 Wasserstein 距离优化拓扑结构。
2. **你借鉴了什么**（1-2 句）：本文沿用了 Hu 提出的 "将持续同调引入分割训练" 的框架，采用 cubical ripser (cripser) 作为 PH 计算后端。
3. **你和 Hu 的差异**（2-3 句）：与 Hu 的监督匹配范式不同，本文采用先验正则化范式——不依赖 GT 拓扑信息，而是通过分析 PD 结构设计惩罚规则。实验发现，PD→loss 的映射方向是拓扑约束成败的关键因素，原始 top-k MSE 映射会鼓励碎片分量持久存在。

### Day 3：提及 clDice 的差异化定位

在相关工作中加一段（3-4 句）：
- clDice 从形态学骨骼（中心线）角度保持拓扑连通性
- 本文从代数拓扑（持续同调）角度约束连通分量结构
- 两者互补：clDice 对空间连续性敏感但不直接控制 Betti 数；PH 直接操作 Betti 数但不感知空间路径

## 5. 答辩可能被问到的 Hu 相关问题 + 准备答案

| 评委可能的问题 | 准备的回答 |
|---|---|
| "你和 Hu 2019 有什么区别？" | "Hu 采用监督匹配范式——比较 pred 和 GT 的持续图。我们采用先验正则化范式——不依赖 GT 拓扑，而是直接惩罚碎片分量。这使得方法更简洁，但也意味着我们需要仔细设计惩罚规则。" |
| "为什么不直接用 Hu 的方法？" | "Hu 的方法需要对 GT 也计算持续同调并做最优匹配，计算复杂度更高。我们选择了一种更轻量的路径——直接对预测端的 PD 施加先验约束。实验表明，关键不在于计算复杂度，而在于映射方向的正确性。" |
| "Hu 的效果比你好吗？" | "Hu 的实验在不同数据集（CREMI 神经元数据）上进行，与我们的视网膜血管任务不直接可比。但从方法论角度，Hu 的监督匹配在有高质量 GT 拓扑时可能更精确，而我们的先验正则化更通用。" |
| "为什么不用 clDice？" | "clDice 从形态学（中心线）角度保持连通性，是一种有效的方法。本文选择从代数拓扑（PH）角度入手，直接操作持续同调结构，目的是探索这条路径的可行性和瓶颈。两种方法互补，可在后续工作中结合。" |
